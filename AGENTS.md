# AGENTS.md

This file is the persistent source of truth for future agents working in this repo. Update it whenever product, design, architecture, tooling, or workflow decisions change.

## 1. Project Overview

Jameel First App is a mobile-first React Native product surface for Jameel Motors in Italy. The August MVP is a guest-only EV Expert Hub focused on pre-sales discovery and evaluation for Geely and Zeekr, with no login, order tracking, or ownership features.

The app opens from a Jameel Motors umbrella Home Page, lets users choose Zeekr or Geely, then moves into a Jameel-owned brand space with four shallow tabs: Home, Product, Network, and Membership. Jameel Motors governs structure, trust, services, support, and network reassurance; Zeekr and Geely provide brand-specific content, models, imagery, accents, and metrics.

Canonical product and design context lives in `PRODUCT.md` and `DESIGN.md`.

## 3. Architecture Principles

- Feature-based modules: product features live under `src/features/<feature-name>` and are driven by `PRODUCT.md`.
- Mock-first services: consumers depend on typed interfaces, not concrete implementations.
- Single backend boundary: the future Nest.js BFF is the only backend the app talks to. No external or third-party service clients are allowed in the mobile app.
- Tokens-only styling: UI code must use `src/theme` tokens for color, spacing, radius, typography, and layout values.
- i18n and RTL from day one: all display strings go through i18next, and directionality is centralized in `src/i18n/rtl.ts`.
- Shape before building: ambiguous product work should capture purpose, user state, content, edge cases, constraints, and success criteria before implementation.

- `__tests__/`: cross-cutting and route-level tests. Feature tests should live beside their feature when features exist.

## 5. Conventions

- Naming: use PascalCase for components and types, camelCase for functions and variables, and kebab-case for feature folders.
- Imports: use `@/...` for `src/...` and `@app/...` for route imports in tests or tooling. Avoid deep relative imports across ownership boundaries.
- Per-feature structure: use this default unless a feature needs less:
  - `src/features/<feature-name>/components`
  - `src/features/<feature-name>/hooks`
  - `src/features/<feature-name>/types.ts`
  - `src/features/<feature-name>/index.ts`
  - `src/features/<feature-name>/README.md`
- Component patterns: keep shared components presentational, token-styled, typed, and unaware of service implementations.
- State rules: use Zustand for local client/UI state only. Use TanStack Query for async/server state, caching, retries, invalidation, and request lifecycle.
- Error handling: services return the normalized `ApiResponse<T>` shape from `src/services/types.ts`. UI-facing error copy must be localized.

## 6. Styling & Theming

- Tokens live in `src/theme/tokens.ts`.
- UI code must not use raw colors, spacing, radii, typography sizes, font weights, or layout dimensions.
- Current app token values are still placeholders. `DESIGN.md` defines the intended system: Jameel Motors as the master structure, Zeekr and Geely as brand color modes, Urbanist typography, 4px spacing scale, white/black CTAs, strict contrast rules, and reusable brand-aware components.
- Before implementing product screens, update `src/theme/tokens.ts` to match `DESIGN.md` and keep future changes synchronized with both files.
- Theme access goes through `ThemeProvider` and `useAppTheme`.

## 7. Internationalization & RTL

- Locale resources live in `src/i18n/locales`.
- Add strings to locale files using stable semantic keys, not English sentence keys.
- Supported locales are declared in `src/i18n/resources.ts`.
- RTL detection and React Native direction sync live in `src/i18n/rtl.ts`.
- The initial market is Italy. Screens must keep language consistent within each locale. The scaffold currently includes English and Arabic placeholder resources; add Italian before product UI work depends on localized copy.

## 8. Services & Data Layer

- All app data access goes through `src/services`.
- `AppServices` is the consumer-facing contract. The current implementation is `mockServices`.
- `ApiClient` defines the future BFF boundary with a configurable base URL via `EXPO_PUBLIC_API_BASE_URL`.
- Concrete endpoints and DTOs remain TODO during feature shaping. The NestJS BFF is future scope and must remain the single backend boundary.
- MVP entities from `PRODUCT.md` include Brand, Vehicle, Service, Dealer, Workshop, Lead, TestDriveRequest, BrandSwitchState, RoutePlannerSession, MembershipItem, and ContentItem.
- To add a new service later:
  1. Add typed DTOs and service interface methods in `src/services/types.ts` or a dedicated service contract file.
  2. Implement the mock in `src/services/mock`.
  3. Export through `src/services/index.ts`.
  4. Consume only through `useServices()` or injected `AppServices`.
  5. Add the BFF implementation behind the same interface later, with no consumer changes.

## 9. Testing Strategy

- Use Jest and @testing-library/react-native.
- Test behavior visible to users and contract boundaries, not implementation details.
- Route-level scaffold tests live in `__tests__`.
- Future feature tests should live beside the feature they cover.
- Service implementations should have contract tests once concrete service methods exist.
